發表於 2016-08-12

本文介绍ceph cache-tier的内部机制，同时对生产环境下提出的一些问题给出解答。
本文是360基础架构组ceph系列第二篇文章。如果你对ceph不太了解，请猛戳[]。

一. 背景：

我们知道容量越大价格越低的存储介质，其读写响应速度就越慢。计算机系统将存储介质由快到慢分级组织起来，逐级访问。这样使得热数据分布在速度更快的上级介质上。提高了平均访问速度。csapp中的存储层次图很好的说明了这点。这里就不贴图了。
同样的目的，ceph引入cache-tier特性，来更好的组织存储介质。一个典型的应用场景是：在一个ceph集群中，为一个存于机械盘的base-pool配置一个存于ssd盘的tier-pool作为其cache-tier，热数据被cache在tier-pool，享受ssd的读写性能。

二. 设置cache tier

设置cache tier很简单，假设你有两个pool tier和base，想将tier作为base的cache。那么只需要执行下面的命令：

ceph osd tier add base tier
ceph osd tier cache-mode tier writeback
ceph osd tier set-overlay base tier

ceph osd pool set tier hit_set_type bloom
ceph osd pool set tier hit_set_count 12
ceph osd pool set tier hit_set_period 14400
ceph osd pool set tier target_max_bytes 1000000000000

我们先来解释前面三条命令的内部实现，后面4条命令与回刷机制有关，我们将在后续讲解.
我们知道，每个pool对应一个pg_pool_t结构体保存其信息，有如下成员：

我们知道，ceph执行设置一个管理命令的内部过程为：客户端将请求发送到monitor集群，monitor集群更新pool对应的pg_pool_t结构体并重新存入osdmap，通过paxos commit这条更改并更新集群epoch，客户端下次访问base-pool时，根据新epoch拉取到最新的osdmap，取出对应pool的pg_pool_t,并是否决定转发到cache-tier pool中。
因此设置cache-tier的代码位于 src/mon/OSDMnitor.cc 中。

1. add base tier

这个命令实际上设置了base-pool的tier字段为tier-pool，设置tier-pool的tier-of字段为base-pool。
这个过程可以看到设置tier有下面几个限制：

  1. base-pool不能是别人的cache tier
  2. tier不能已经是别人的tier 这两条是为了防止循环写入转发
  3. tier-pool 必须是一个空pool，这是为了防止误读
  4. tier-pool不能是ec编码的pool，我们知道ec编码是牺牲响应时间来提高存储空间效率的方案，而cache tier则放弃牺牲空间存储效率来提高响应时间，这两个是不相容的。

这些判断逻辑能够防止对内部不够了解的用户误用cache-tier。我觉得这是很好的设计思路。

##二.构造函数语义

构造函数分为default constructor及 copy constructor

###1. default constructor

  • member class：当member class有默认构造函数时，会构造外围class的默认construtor或在其constructor执行前插入行为，按声明顺序执行member class的初始化。
  • base class:同member class 行为一致，编译器构造non-trival的子类constructor
  • virtual function: 生成一个non-trival的constructor ，使vptr指向vtbl以实现多态
  • virtual base class：生成non-trival的constructor，生成指向virtual-base-class的指针
    结论
  1. 以上4种情形生成non-trival constructor class
  2. 除去有构造函数的member class及base class，类的其他非static成员并不会被初始化，如int，int* 等

###2. copy constructor

  • explicit ：声明形如 A::A(const A&a)的拷贝构造函数后，下面4种情况编译器都会调用拷贝构造函数。

    A a1;
    A a(a1);
    A a = a1;
    A a = A(a1);

  • implicit：在上面的情况下A a = b (class A) 时，如默认的member class构造一般，会依次将a成员赋值为b成员值。这叫做memberwise initialization.此时，被赋值的成员包括：int等基础类型，有默认构造函数的member class

###3. 程序语义

  • c++函数如何传入及返回一个class？

    传参实际上调用拷贝构造函数先构造一个参数，传入该引用。
    返回参数实际上传入一个额外的引用，而在最终调用拷贝构造函数将该引用所值内容赋成return value的值。
    那么一个函数 A foo()实际变为:
    void foo(A& a);

ceph基本介绍

發表於 2016-05-01

一. ceph项目概述

ceph是一套强一致的分布式存储系统。它主要有两个特点，

  1. 其通过对象存储包装出块存储，posix文件存储接口。因此ceph支持对象/块/posix文件存储
  2. 其支持client到存储端直接访问(通过crush算法)而无需代理。

其技术栈如下图所示。

1.1 ceph架构及几个主要流程简介

ceph后端存储集群的设计思路特别主流，主要有三点：

  • 有中心节点

    几个monitor节点组成集群，通过paxos强一致的管理元信息。

  • 主从结构

    placement是管理数据基本单元，几个placement 按一主多从组成placement group(pg)，每个pg存储多个对象，每次读写请求均打向主。从作为备份，当主挂掉后接管pg服务。ceph是强一致系统，因此每次写，主确认两个从写入成功后才返回。

  • 一致性hash

    osd是实际存储节点，每个osd上面有多个placement。每个pg里的pg分布在不同的osd上。这样当osd数量改变后，ceph只需要对其涉及到的部分pg进行操作，迁移扩容都很方便。

1.2 读写流程简介

####客户端

1.客户端librados 从monitor拉取最新的clustermap
2.客户端hash key 得到pg数
3.客户端根据从；clustermap和crush算法得到pg对应 osd向其发送请求

####ceph集群

  1. 主osd 收到请求，写本地同时向其他从写
  2. 从分别写本地并向主返回成功
  3. 主确认所有从写成功后返回成功

1.3 scrub机制简介

  • scrub 检测主从中对象元信息的一致性，发现数据不一致后，可以通过ceph pg repair pg_id 修复（覆盖从）

  • deep-scrub 检测主从中对象数据一致性，默认一周一次

1.4 数据恢复的机制简介

a. osd节点临时挂掉后，通过peering恢复数据。peering过程时间

b. 新加入osd节点,或永久删除某osd需要将pg迁移到其他osd时通过backpill 全量拷数据。
backfill可能将磁盘跑满，影响线上服务。限制backfill流量的参数配置：

osd-max-backfills 1
osd-max-recovery-threads 1
osd-recovery-op-priority 1
osd-client-op-priority 63
osd-recovery-max-active 1

1.5 heartbeat机制简介

  • 每个osd与与其有交叉pg的osd集合进行心跳,heartbeat失败则汇报monitor
    心跳间隔参数: osd heartbeat interval
    心跳超时参数: osd heartbeat grace

4.1 TCMALLOC问题的解决

现象：

sudo perf top -p pid 发现tcmalloc 占用高

原因：

ceph 0.87.2 与 机器上较低版本的 tcmalloc 兼容性有问题

解决方案有两个：

  • 将ceph内存管理换为jemalloc
  • 提高本地tcmalloc版本,同时调高max_bytes(export max_bytes=128m)
    ####方案一实现

    1.编译ceph
    sudo yum install libunwind-devel.x86_64
    ./configure –without-tcmalloc and –with-jemalloc
    make
    2.替换 /usr/bin/ceph-osd
    3.安装jemalloc

方案二实现

1.编译gperftool
./configure –enable-frame-pointers
make

  1. 修改环境变量并重启
    export TCMALLOC_MAX_TOTAL_THREAD_CACHE_BYTES=128M
    restart osd

磁盘监控

a. vmstat

vmstat时iowait是什么意思？

每次clock定时中断上cpu后，统计之前cpu状态，用户态则user加1，内核态则sys加1，idle且无io等待则idle加1，idle且有io等待则iowait加1。可见iowait可以用来判断cpu是否是瓶颈，低iowait说明cpu一定不是瓶颈。而用iowait看磁盘瓶颈则不太靠谱。

b. iostat :iostat -x 1

  • iostat的util是什么意思？
    每次tick采样，device 在进行io操作的比例

  • iostat的await，svim什么意思？
    每次io操作，现在队列等待，然后进行操作，操作时间svim，总时间为await

因此util高负载不一定很高，因为并行的缘故，衡量device忙碌程度一般用svim靠谱点
